## Naming Conventions

**Files:**
- Snake case with `.py` extension: `alpha_pan.py`
- Model checkpoints: `model_{iteration}.pt`, `optim_{iteration}.pt`

**Classes:**
- PascalCase: `Chenapan`, `AlphaPanNet`, `Node`, `MCTS`, `AlphaPan`

**Methods:**
- Snake case: `get_initial_state()`, `get_next_state()`, `check_valid_moves()`, `is_fully_expanded()`
- Getters: `get_*` prefix (e.g., `get_encoded_state()`, `get_opponent()`)
- Checkers: `check_*` prefix for validation (e.g., `check_win()`, `check_ace_moves()`)

**Variables:**
- Snake case: `valid_moves`, `state`, `action_probs`, `value_sum`
- Constants: UPPERCASE: `INITIAL_BOARD`, `VALID_MOVE_MATRIX`, `MAX_NUMBER_OF_MOVES`
- Single letters for iteration/loop: `i`, `j`, `row`, `column`

**Functions:**
- Snake case: `flatten_and_sum_list_of_list()` (utility function at module level)

## Where to Add New Code

**New Feature (e.g., endgame database):**
- Primary code: Add method to `Chenapan` class (lines 84-390 in `alpha_pan.py`)
- Supporting logic: Add to `AlphaPanNet` if affecting network input/output
- Tests: None currently - would create as separate test functions or Jupyter cells

**New Component/Module:**
- Implementation: Currently monolithic in `alpha_pan.py` - would split into separate files at root
- Convention: Create `game.py`, `model.py`, `mcts.py`, `trainer.py` as modules
- Example structure:
  - `game.py`: Chenapan class
  - `model.py`: AlphaPanNet class
  - `mcts.py`: MCTS, Node classes
  - `trainer.py`: AlphaPan class
  - `train.py`: Entry point script

**Utilities:**
- Shared helpers: Add functions at module level in `alpha_pan.py` (like `flatten_and_sum_list_of_list()` at lines 77-82)
- Convention: Prefix with underscore if internal-only (e.g., `_validate_move()`)

**Move Validation Logic:**
- Per-piece implementation: Use `check_*_moves()` pattern (lines 193-332)
- New piece type: Create method `check_newpiece_moves(state, row, column)` in Chenapan
- Update dispatcher in `get_valid_moves()` (lines 146-181) to call new method

**Game Constants:**
- Location: Top of `alpha_pan.py` (lines 24-75)
- Pattern: Module-level, UPPERCASE names, include explanatory comments in French

## Monolithic Structure Notes

Current codebase is a single 932-line Python file. For refactoring into modules:

1. **Separate concerns into files:**
   - `alpha_pan/game.py`: Chenapan, VALID_MOVE_MATRIX, INITIAL_BOARD
   - `alpha_pan/model.py`: AlphaPanNet
   - `alpha_pan/mcts.py`: MCTS, Node
   - `alpha_pan/trainer.py`: AlphaPan, selfPlay, train methods
   - `alpha_pan/config.py`: args dictionary and hyperparameters
   - `alpha_pan/train.py`: Entry point

2. **Maintain imports:**
   - Entry point imports from all submodules
   - Each module imports numpy, torch, and dependencies as needed
   - Avoid circular imports by using dependency injection (pass game, model to classes)

3. **Constants migration:**
   - Move VALID_MOVE_MATRIX, INITIAL_BOARD, MAX_NUMBER_OF_MOVES to game.py
   - Move args dict to config.py
   - Import in train.py
